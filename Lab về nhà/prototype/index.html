Design Rationale — T0 (10 câu)

  • AI biết: Chưa gì về user
  • AI chưa biết: Style, ngân sách, quyền dữ liệu user cho phép
  • Giả định: User muốn lập lịch nhanh, không muốn form dài
  • Có thể sai: User tưởng AI tự đặt phòng/thanh toán
  • Hậu quả: Thất vọng, mất niềm tin — user chịu ảnh hưởng trực tiếp
  • Phát hiện/hoàn tác: Dễ — chưa có hành động nào để hoàn tác
  • Don't Act: Không thu thập dữ liệu khi user bấm "Bỏ qua"
  • User kiểm soát: Bỏ qua quyền, chọn style, rút quyền sau (Cài đặt)
  • Feedback: Chọn style = explicit user; bỏ qua perm = implicit (chưa sẵn sàng chia sẻ)
  • Tiếp tục mục tiêu: 3 bước ngắn → vào lập kế hoạch ngay, không mắc kẹt onboarding

Design Rationale — T2 (10 câu)

  • AI biết: Điểm đến = Đà Nẵng
  • AI chưa biết: Số ngày, người đi, ngân sách, ưu tiên
  • Giả định: 4 thông tin trên đủ để tạo lịch khả dụng
  • Có thể sai: Act ngay → lịch lệch nhu cầu; hỏi quá nhiều → user bỏ dở
  • Hậu quả: User phải sửa nhiều hoặc bỏ app — tốn thời gian chuyến đi
  • Phát hiện/hoàn tác: Lịch nháp — sửa/hoàn tác dễ trước khi xác nhận
  • Ask thu thập; Act chỉ sắp xếp route (rủi ro thấp)
  • User kiểm soát: Chọn chip từng câu, có thể quay lại
  • Feedback: Explicit system = progress + "vì sao cần"; implicit = chip thay form
  • Tiếp tục mục tiêu: 4 câu tối đa → không mắc kẹt, vào chọn phương án
